# Frame 操作指南（給非技術角色）

這份指南帶你完成 Stage 1：Frame。預期投入 30–90 分鐘。完成後你會得到一份 `system-brief.md`，內含十個欄位都填好的系統定錨檔案，可以直接交給 Stage 2 Decompose 使用。你不需要懂程式、不需要懂 schema，只需要照著一格一格填。如果某一格真的填不出來，把它寫進 `openQuestions`，不要硬填。

## 你需要先準備什麼

開始之前，把下面這些東西放在手邊（紙筆、文件、聊天紀錄都可以）：

- 對這個系統的口頭描述：你會怎麼跟同事 30 秒解釋它？
- 預期使用者：腦中浮現的前 1–3 個角色，以及他們會在什麼情境下用這個系統。
- 最近一次討論裡浮現的痛點：把對話紀錄、會議筆記、Slack 訊息找出來。
- 可能的限制：公司既有系統、法規、預算、時程、人力。
- 你心裡那條「萬一這個假設是錯的就完蛋」的句子。

## 一步一步填

### oneLiner — 一句話定義系統

- **要填什麼**：用一句話告訴別人這是什麼系統、給誰、解決什麼。
- **填寫範本**：
  - zh-TW 範本：`這是一個 ___，幫助 ___ 解決 ___。`
  - en 範本：`This is a ___, helping ___ solve ___.`
- **範例**：「這是一個個人讀書清單系統，幫助長期想讀很多書的人解決『書買回家後永遠沒讀』的痛點。」
- **小心**：避免寫成「一個強大的、整合的、AI 驅動的平台」這種空話；空話越多，後續每一階段就越會在錯誤目標上累積。

### problem — 我們嘗試解決的痛點

- **要填什麼**：列 1–5 條此系統正面對決的痛點，每條一句話。
- **填寫範本**：`<誰> 在 <什麼情境> 下，因為 <原因> 而 <受到什麼影響>。`
- **範例**：「想養成閱讀習慣的上班族，因為書單分散在 Amazon、Notion、實體便利貼上，常常買重複或想讀時找不到。」
- **小心**：寫的是「痛點」不是「功能需求」。「需要一個 API」不是痛點。

### nonProblems — 我們明確不解決的事

- **要填什麼**：列 1–5 條看起來相關、但這個系統不打算碰的事。
- **填寫範本**：`不解決 <某項問題>，因為 <理由>。`
- **範例**：「不解決電子書閱讀體驗，因為 Kindle 已經做得夠好。」
- **小心**：這一格越具體，後續越不會被「順便加一下」式的需求拖垮。

### targetUsers — 目標使用者

- **要填什麼**：1–5 個角色，每個角色搭配使用情境。
- **填寫範本**：`<角色名稱>，在 <情境> 中使用本系統。`
- **範例**：「重度讀者，在通勤搭捷運時想決定接下來看哪一本書。」
- **小心**：「所有使用者」不是角色；角色必須具體到能想像他打開系統時的姿勢。

### successSignals — 可量測的成功訊號

- **要填什麼**：3–5 條我們敢說「做到這個就是成功」的訊號。每一條都要可量測。
- **填寫範本**：`<指標> 從 <現況> 變成 <目標> 在 <時間範圍內>。`
- **範例**：「個人書單『買了沒讀』的比例從 70% 降到 30%，三個月內。」
- **小心**：避免使用 `better / faster / 更好 / 更快 / 提升` 這類詞——Vector wizard 的 `validateDraft` 規則會直接擋下這種句子，因為它們無法被驗證。

### uniqueValue — 獨特價值

- **要填什麼**：為什麼是我們做、而不是直接買現成工具或繼續用試算表。
- **填寫範本**：`相較於 <現有方案>，本系統提供 <獨特之處>。`
- **範例**：「相較於 Goodreads，這個系統把『要讀的書』與『實體書放哪一格』綁在一起，避免找不到書。」
- **小心**：如果寫不出比現成工具強的地方，先停下來——也許真的不該做。

### keyMetrics — 關鍵指標

- **要填什麼**：1–5 條上線後會盯著看的指標，每條附目標值。
- **填寫範本**：`<指標名稱>：<目標值>。`
- **範例**：「每週主動開啟系統的天數：≥ 3 天。」
- **小心**：指標必須能被量到（不是「使用者滿意度」這種無法直接觀察的東西）。

### constraints — 硬限制

- **要填什麼**：技術、法規、組織上不能違反的條件。
- **填寫範本**：`必須 <條件>，因為 <原因>。` 或 `不得 <行為>，因為 <原因>。`
- **範例**：「必須能離線使用，因為通勤地下化區段沒網路。」
- **小心**：法規類限制（個資、GDPR、財報合規）最容易漏；漏了會在後期整個重做。

### riskiestAssumptions — 最危險的假設

- **要填什麼**：1–3 條若被推翻，整個系統就不成立的假設。
- **填寫範本**：`我們假設 <某事為真>；若實際上 <某事為假>，則本系統不應被建造。`
- **範例**：「我們假設使用者會主動每週開系統一次；若使用者只在週末有空、且週末通常忘記打開，則本系統不應存在。」
- **小心**：若你寫完一條讀過去毫不緊張，這條就不夠危險，再寫一條。

### openQuestions — 還沒答案的問題

- **要填什麼**：現在答不出來、但需要在後續階段追問的問題。
- **填寫範本**：`<問題>？需要 <誰/什麼方式> 來回答。`
- **範例**：「實體書放哪一格的資料，是否需要照片辨識？需要做一次使用者訪談來確認。」
- **小心**：寧可寫得太多，也不要把模糊偽裝成確定。

## 完成檢查表

填完十個欄位後，逐項打勾：

- [ ] 十個欄位都有非空內容，沒有「待補」或佔位文字。
- [ ] `successSignals` 每一條都是可量測句，沒有出現 `better / faster / 更好 / 更快 / 提升`。
- [ ] `riskiestAssumptions` 至少有一條讓你讀完會心痛、會緊張。
- [ ] `openQuestions` 至少有 1 條，不是空陣列。
- [ ] 把整份檔案唸出來給沒參與過討論的人聽，他能在 5 分鐘內回答「這是什麼系統？為誰？什麼算成功？」。

## 常見坑

1. **`nonProblems` 寫成 vague 形容詞**：例如「不解決 UX 問題」——這句沒有邊界。改成「不解決多人協作編輯，因為單機體驗已足夠」。
2. **`successSignals` 是動作而不是結果**：例如「做完登入功能」是動作不是結果。改成「使用者首次登入到看到儀表板少於 30 秒」。
3. **`constraints` 漏掉法規層**：個資、GDPR、PCI、財報合規這類限制最容易漏；填寫前主動問一次「有沒有任何法規要符合？」。
4. **`riskiestAssumptions` 寫得太安全**：若這條被推翻你只會說「喔好可惜」，那就不是最危險的；繼續逼自己寫到一條讀完會反胃為止。
5. **`targetUsers` 寫成「所有人」**：這代表你還沒想清楚使用情境；改成兩個具體的角色 + 對應情境，即使只有兩個也比一個「所有人」有用。
