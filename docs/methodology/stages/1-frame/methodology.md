# Stage 1：Frame（系統定錨）

在開始切割任何功能、寫下任何 user story 之前，我們先把「我們究竟在做什麼系統」用文字寫下來，並由所有相關角色一起讀過一次。這個動作看起來像形式主義，但實際上是後續所有切割的座標原點：若沒有書面定錨，每一次切割都會在不同的、口耳相傳的目標上累積，最後產出的 backlog 看起來合理，但拼起來不是任何人想要的系統。Frame 階段的產物只有一份檔案 `system-brief.md`，但它必須能讓任何拿起它的人（包含三週後的我們自己）在 5 分鐘內回答「這個系統是什麼？為誰做？什麼算是成功？」。

## 方法論組合

Frame 不發明新的方法論，而是把三個成熟工具裁剪後組合起來，覆蓋彼此的盲區。

- **Problem Statement（問題陳述）**：來自設計思考與 lean startup 的標準起手式，要求我們用一兩句話寫下「為誰、解決什麼、為什麼現在」。在 Frame 中它對應 `oneLiner` 與 `problem` 兩個欄位，是其他所有欄位的錨。
- **Lean Canvas（精簡版）**：Ash Maurya 的九宮格模型本身偏向商業面，我們只取其中與系統定錨直接相關的格子：問題、目標客戶、獨特價值、關鍵指標。商業面的成本結構、收益模式、通路在這個階段刻意不放進來，避免把「定義系統」變成「寫商業計畫」。
- **Hypothesis-driven Development（假設驅動開發）**：Marty Cagan 與 Jeff Patton 都強調，每個產品決策背後都有一條我們害怕被推翻的假設。Frame 把這條假設明確寫進 `riskiestAssumptions`，逼我們承認：這個系統能不能成立，最終取決於哪一條我們現在還沒驗證的事。

## 為何選擇這三者

任何單一方法論獨用都會留下明顯破洞。Lean Canvas 完整版資訊量過大，逼我們在還沒搞清楚系統前就要回答商業模式，討論常常卡在這裡而沒人真的去寫第一段功能。Problem Statement 的格式又太單薄，只描述問題本身，沒有逼我們承認「我們可能搞錯了什麼」，於是團隊容易在錯誤的問題上一路衝刺。Hypothesis-driven 補上的正是這個破洞——它要求我們把「這個系統能成立的前提」寫成一條可被推翻的句子，而不是一個美好的願景。三者組合起來：Problem Statement 抓住起點、Lean Canvas 抓住周邊、Hypothesis-driven 抓住風險。我們不追求完美，只追求把這三個視角各自最關鍵的一句話寫下來。

## 十個必填欄位

| 欄位 | 用途 | 若略過的代價 |
| --- | --- | --- |
| `oneLiner` | 一句話定義系統，給所有後續討論一個共同錨 | 三週後沒人記得當初到底在做什麼 |
| `problem` | 列出 1–5 條我們嘗試解決的核心痛點 | Decompose 階段切出與痛點無關的能力 |
| `nonProblems` | 明確寫下「我們不解決什麼」 | 範圍蔓延，每次討論都要重新爭辯邊界 |
| `targetUsers` | 1–5 個角色 + 使用情境 | user story 主詞變成模糊的「使用者」 |
| `successSignals` | 可量測的成功訊號（不是「更好/更快」） | 完工後沒人能宣告「做到了」 |
| `uniqueValue` | 為什麼是我們做、而不是現成工具 | 做到一半發現可以直接買 SaaS |
| `keyMetrics` | 1–5 條會盯著看的指標 + 目標值 | 上線後不知道要看什麼數字 |
| `constraints` | 技術、法規、組織硬限制 | 切到一半才發現要符合 GDPR、整個重來 |
| `riskiestAssumptions` | 1–3 條最害怕被推翻的假設 | 賭最大的一把卻沒人意識到那是賭 |
| `openQuestions` | 現在還答不出來、但需要追問的問題 | 把模糊偽裝成確定，後續放炸彈 |

## 與下游的對應

Frame 不是孤立文件。它的內容會在後續階段以結構化形式延續下去：

- `successSignals` 直接複用為 Stage 4 feature-seed 的 `goal.successSignals`，因此這裡寫的訊號必須是個別 feature 也能套用的句型。
- `constraints` 與 `riskiestAssumptions` 在 Stage 4 會分別下沉到每個 feature 的 `agentBoundaries.constraints` 與 `agentBoundaries.risks`，AI 編碼代理會直接讀到這些限制。
- `openQuestions` 不會消失，它會以 `agentBoundaries.openQuestions` 的形式被帶進每個 feature-seed，提醒實作者：這條路徑還沒拍板、若答案改變則整個 feature 要重新評估。

換句話說，Frame 是後續所有檔案共用的「上文」。在這裡偷懶的代價會在 Stage 4 之後以 N 倍放大。

## 何時不適用

Frame 並不適合每天都在推翻假設的早期探索。如果我們連「使用者是誰」都還在每週改、連解決的問題都還沒收斂，就把十個欄位填滿只會產出虛假的確定感。這種情境屬於 Path C（探索性 prompt）：先用 ChatGPT 之類的工具做幾輪自由對話、丟出幾個原型，等我們能口頭說出穩定的 oneLiner 之後，再回到 Path B 走 Frame。Frame 服務的是「我們已經有共識想要做什麼，只是還沒寫下來」的情境，不是「我們還在尋找想做什麼」的情境。

## 進入下一階段的條件

進入 Stage 2 Decompose 之前，我們會用三個檢查確認 Frame 真的完成了，而不是只是把欄位填滿：

1. 十個欄位都有非空、具體的內容；沒有任何欄位寫著「待補」或佔位文字。
2. `riskiestAssumptions` 至少有一條讓我們讀完會緊張、會想去驗證；如果讀完毫無感覺，代表寫得太安全、不是真正的最危險假設。
3. `openQuestions` 不是空陣列；坦承未知比假裝什麼都知道更能保護後續階段。

只要這三項通過，Frame 就足夠把錨打下。它不需要完美，但需要誠實。
